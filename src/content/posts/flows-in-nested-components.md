참고로 클린업 함수는 컴포넌트가 언마운트될때뿐만 아니라 디펜던시가 변경되어 유즈이팩트의 콜백을 다시 실행하기 전에도실행됨. 왜냐? 디펜던시가 변경되면 부모컴포가 리렌더링되기때문.

각 프로세스에 이유를 추가해보기

# 자식 컴포넌트 추가

1. 부모 컴포넌트의 모든 코드 순서대로 실행
2. 자식 컴포넌트의 모든 코드 순서대로 실행
3. 부모 컴포넌트의 useEffect의 클린업 함수들 실행 (해당하는 데이터)
   1. 배열없는애랑 변경된 state (자식 렌더링을 트리거하는애)
4. 자식 컴포넌트의 useEffect 콜백 순서대로 실행 (모두)
5. 부모 컴포넌트의 useEffect 콜백 순서대로 실행 (해당하는 데이터)
   1. 배열없는 애랑 변경된 state (자식 렌더링을 트리거하는애)

# 자식 컴포넌트 제거

1. 부모 컴포넌트의 모든 코드 순서대로 실행
2. 자식 컴포넌트의 useEffect의 클린업 함수들 실행 (모든 클린업 함수)
3. 부모 컴포넌트의 useEffect의 클린업 함수들 실행 (해당하는 데이터)
   1. 배열없는 애랑 변경된 state (자식 렌더링을 트리거하는애)
4. 부모 컴포넌트의 useEffect 콜백 실행 (해당하는 데이터)
   1. 배열없는 애랑 변경된 state (자식 렌더링을 트리거하는애)

# 자식 컴포넌트 업데이트

1. 자식 컴포넌트의 모든 코드 순서대로 실행
2. 자식 컴포넌트의 클린업 함수 실행 (해당하는 데이터)
   1. 배열없는애랑 업데이트된 state
3. 자식 컴포넌트의 useEffect 콜백 실행 (해당하는 데이터)
   1. 배열없는애랑 업데이트된 state
